## 🎯 Мета
Зараз живий режим прив'язаний до НАШОГО Ko-fi (один ключ `pixelkava`). Мета v3: **кожен стрімер** вмикає живі донати зі **своїм** Ko-fi, а обслуговує всіх **один наш Worker**. Дані кожного — окремо, за унікальним `id`.

## 🧠 Ключові поняття (простими словами)
- **`id`** — унікальний код стрімера (напр. `a1b2c3d4e5f6`, 12+ випадкових символів). Іде в посилання віджета і в webhook-URL. Публічний — підглянути не страшно (сума й так публічна).
- **`cfg:<id>`** — приватний конфіг стрімера в KV: його Ko-fi-токен + його ключ скиду. **Ніколи не віддається назовні.**
- **`goal:<id>`** — лічильник стрімера (`{current, goal}`).
- **resetKey** — окремий випадковий секрет стрімера для `/reset` (щоб тільки він міг скинути).

## 🗄️ KV-розкладка
| Ключ | Значення | Доступ |
|---|---|---|
| `goal:<id>` | `{current, goal}` | читання — публічне (`/total`) |
| `cfg:<id>` | `{token, resetKey, created}` | лише всередині Worker |
| `pixelkava` *(legacy)* | наш власний лічильник | сумісність зі старим |
| `rl:<ip>` *(rate-limit)* | лічильник реєстрацій/час | антиспам |

## 🔌 Ендпоінти Worker
| Метод/шлях | Що робить | Безпека |
|---|---|---|
| `POST /register` `{token}` | генерує `id`+`resetKey`, пише `cfg:<id>`, ініціалізує `goal:<id>={0,100}`; повертає `{id, widgetUrl, webhookUrl, resetUrl}` | **rate-limit за IP**, валідація формату токена, без особистих даних |
| `POST /webhook?id=<id>` | бере `cfg:<id>.token`, звіряє `payload.verification_token`, додає суму в `goal:<id>` | 403 при невірному токені; 200 на порожні пінги |
| `GET /total?id=<id>` | віддає `goal:<id>` (CORS `*`) | лише читання; дефолт `{0,100}` |
| `GET /reset?id=<id>&key=<resetKey>[&goal=N]` | скид лічильника стрімера | 403 без вірного `resetKey` |
| *(без `id`)* | falls back на `pixelkava` | сумісність із v2 |

## 🖥️ Сторінка «Підключити Ko-fi» (фаза 2, публічна)
Статична сторінка на GitHub Pages (5 мов), яка:
1. Показує **пояснення для новачків** (як це працює, 3 кроки) — усіма мовами.
2. Поле: вставити свій **Ko-fi Verification Token** → кнопка «Підключити» → `POST /register`.
3. Показує результат: готове **посилання віджета** (`goal.html?...&api=…/total?id=ID`), **webhook-URL** (`…/webhook?id=ID`), **закладку скиду**, і крок-за-кроком як вставити webhook у Ko-fi та віджет в OBS.
4. Безпека: CSP (+`connect-src`), `textContent`, без логів токена, HTTPS.

## 🌍 Мови
Канонічні **EN · UK · ES · PT · DE** (стандарт §14.1 OPERATING_CONTEXT): автовизначення браузера + пам'ять + fallback EN. 🚫 без російської. Сама сторінка «Підключити» та пояснення — усіма 5.

## 🔐 Безпека (пройти перед фазою 2)
- Ko-fi `verification_token` — це **не** доступ до грошей/акаунта; максимум — хтось із чужим id+токеном міг би накрутити *чужу* смужку. Усе одно тримаємо як секрет: у KV, **ніколи в логах/відповідях GET**.
- `id` і `resetKey` — **випадкові, невгадувані** (crypto random).
- **Rate-limit** реєстрацій (антиспам KV).
- Реєстрація = **тільки створення нового** (новий випадковий id щоразу) → не можна перезаписати чужий конфіг.
- **Нуль особистих даних** (без email/імені) — privacy-first.
- Валідація вводу токена (формат/довжина), CSP, `textContent` (анти-XSS).

## 🗺️ Фази
- **Фаза 1 (зараз, непублічно):** переписати Worker на `id`-логіку (`goal:<id>`/`cfg:<id>`, `/webhook`,`/total`,`/reset` з `id`, fallback `pixelkava`) + `/register` із rate-limit. Код локально + комміт; **деплой — за згодою капітана (§8.5).**
- **Фаза 2 (окрема згода, §8.5):** публічна сторінка «Підключити» (5 мов + пояснення новачкам) + безпековий чеклист.
- **Фаза 3:** конструктор підставляє `id` у живий режим автоматично (після реєстрації).

Інший стрімер може: зайти → вставити свій Ko-fi-токен → отримати готове посилання → донати йдуть у ЙОГО смужку наживо → скинути перед стрімом своїм ключем. Без коду, без особистих даних, 5 мов.
